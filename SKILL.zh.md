# 反思式解决方案框架

```
评估 → 方案 → 审视 → 迭代
```

## 何时使用

当用户：
- 要求分析或审查代码/实现
- 问"这种做法有什么问题"
- 问"你准备怎么解决"
- 问"你觉得你的方案怎么样"
- 要求评估某个工作流程

## 核心循环

### 第一步：评估
- 分析当前实现或情况
- 识别潜在问题（边界情况、竞争条件、错误处理）
- 理解用户的真实需求和约束

### 第二步：方案
- 给出具体可行的解决方案
- 说明实现思路
- 指出适用场景

### 第三步：审视（关键）

**必问问题：**
1. 我的方案解决了根本问题吗，还是只打补丁？
2. 有什么边界 case 没考虑到？
3. 有什么更好的设计思路？
4. 这个方案适合什么场景，不适合什么场景？
5. 有没有更简单、更安全的替代方案？

**审视维度：**
- 可靠性（并发、失败恢复）
- 性能影响
- 安全性和数据完整性
- 维护成本
- 用户体验

### 第四步：迭代
- 基于审视结果调整或提出新方案
- 如果发现更好的思路，果断替换
- 如果原方案最优，明确说明原因

## 回复格式

```
## 评估
<现状分析>

## 方案
<解决方案>

## 审视
### 优点
- ...

### 缺点 / 边界情况
- ...

### 更好的思路
<如果有的话>

## 结论
<最终推荐>
```

## 反模式

1. 不要急于给答案 - 先完整评估问题
2. 不要只提优点 - 必须指出缺点和边界情况
3. 不要执着于第一个方案 - 审视发现问题就果断换
4. 不要回避"打补丁"的评价 - 诚实指出

## 提示

- 如果问题涉及**数据安全/不可逆操作**，优先考虑最安全的方案
- 如果问题是**性能优化**，提供 benchmark 建议
- 如果用户没有明确场景假设，主动询问或覆盖多种场景
